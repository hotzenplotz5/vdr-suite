It does not replace it.\n\nVDR-Suite should prefer selective backend queries over full-domain transfers whenever possible.\n\nThe runtime performance target is backend workload comparable to established VDR frontends such as `live` when equivalent user-visible information is requested.\n\n---\n\n## Current Position\n\n```text\nCompleted implementation state\nPhase 43.1 - Recording Action Execution Result JSON Serializer\n\nCurrent cleanup\nDocumentation synchronization after Phase 29.2\n\nNext implementation step\nPhase 36.0 - Recording Action Execution Result JSON Serializer.\n\nCompleted foundation summary\n```\n\nPhase 16 completed the multi-backend polling and runtime context foundation. Phase 24 forward-looking roadmap blocks.\n\nThose earlier roadmap blocks now describe completed implementation foundation or superseded direction. Completed implementation detail belongs in [Completed Phases](../development/completed-phases.md), while this file describes the next architecture direction after Phase 24.5.\n\nThe mid-term roadmap is organized around the following planned phase groups:\n\n```text\nPhase 25.x - EPG REST API Boundary\nPhase 26.x - EPG Startup Decoupling and JSON Cleanup\nPhase 27.x - Backend Optional Runtime and Capability-Aware API Foundation\nPhase 28.x - Recording Query Architecture\nPhase 29.x - Recording REST API Modernization\nPhase 30.x - Recording Actions\nPhase 31.x - Recording Metadata and Job Integration\nPhase 32.x - Multi-VDR Federation\nPhase 33.x - Permission and Role System\nPhase 34.x - Advanced Search\nPhase 35.x - SearchTimer Architecture\nPhase 36.x - Windows Client API Stabilization\n```\n\nThe following rules guide all phase groups:\n\n- VDR remains the source of truth for channels, timers, EPG, running events, real recordings and device state.\n- VDR-Suite does not persist a full EPG mirror in SQLite.\n- EPG remains a heavy domain and must be read selectively.\n- SQLite is reserved for VDR-Suite-owned application data such as recording metadata, jobs, workflow state, user configuration, permissions, roles and optional search indexes.\n- SearchTimer work remains out of scope until the EPG and recording API boundaries are stable.\n- Multi-VDR and permission work must build on backend-neutral APIs.\n- Frontend-facing APIs must remain backend-aware and client-independent.\n\n---\n\n## Phase 25.x - EPG REST API Boundary\n\nGoal:\n\nExpose selective EPG query behavior through a stable REST API boundary for future clients.\n\nPlanned direction:\n\n- add an EPG REST boundary above `IEpgQueryService`\n- preserve backend-neutral EPG query contracts\n- expose Now/Next queries through the REST layer\n- expose time-window EPG queries through the REST layer\n- expose channel-window EPG queries through the REST layer\n- keep query construction outside frontend code\n- keep RESTfulAPI-specific query details inside the adapter layer\n- avoid persistent EPG mirroring in SQLite\n- avoid EPG full-text search in this phase\n- avoid SearchTimer architecture in this phase\n\nRESTfulAPI integration rule from Phase 24.5:\n\n```text\nNow/Next       -> chevents=2\nTimeWindow    -> from + timespan\nChannelWindow -> from + timespan + limit\nonly_count    -> not a fachlicher EPG count path for now\n```\n\nArchitecture rule:\n\n```text\nEPG REST API -> IEpgQueryService -> VdrService -> IVdrAdapter -> backend-specific adapter\n```\n\nThe REST API must expose selective EPG reads without making full EPG refresh the default runtime strategy.\n\nOut of scope:\n\n- SearchTimer support\n- EPG full-text search\n- persistent EPG mirror tables\n- client-side RESTfulAPI URL construction\n- treating `only_count=true` as reliable fachlicher EPG count semantics\n\n---\n\n## Phase 26.x - EPG Startup Decoupling and JSON Cleanup\n\nGoal:\n\nKeep EPG outside blocking daemon startup paths and harden the EPG REST boundary.\n\nImplemented direction:\n\n- remove full EPG loading from the initial runtime snapshot\n- keep Events / EPG outside the initial daemon startup path\n- validate selective EPG query behavior against a real VDR\n- keep EPG available through query-oriented REST APIs\n- harden EPG JSON serialization and escaping\n- avoid premature shared JSON infrastructure while duplication remains local\n\nArchitecture rule:\n\nEPG remains a query domain. The daemon must not require full EPG data before HTTP APIs can become available.\n\n---\n\n## Phase 27.x - Backend Optional Runtime and Capability-Aware API Foundation\n\nGoal:\n\nAllow VDR-Suite to start as an empty runtime platform and prepare client-safe capability-aware behavior.\n\nImplemented foundation:\n\n- daemon runtime can initialize without a configured VDR backend\n- backend polling is skipped when no backend runtime contexts exist\n- EPG service wiring is optional when no backend-backed EPG service exists\n- EPG REST routes report backend unavailability with HTTP 503 instead of crashing runtime initialization\n- backend registry and REST visibility support empty backend lists\n\nImplemented continuation:\n\n- capability state domain model\n- capability resolver state exposure\n- capability state and report JSON serialization\n- capability report builder and service layer\n- capability controller\n- ApiRouter integration for `/api/vdr/capabilities`\n- DaemonRuntime capability controller wiring\n- real VDR validation through daemon HTTP port `18080` and RESTfulAPI backend port `8002`\n\nPlanned direction:\n\n- define backend capability models for channels, timers, recordings, EPG, live transport and actions\n- expose backend capabilities through stable REST contracts\n- make frontend behavior capability-aware instead of platform-specific\n- keep optional backend domains explicit and visible to clients\n- prepare capability-aware routing and dashboard summaries\n\nArchitecture rule:\n\nClients must be able to connect to the API before a backend is configured or available. Backend-specific domains must be represented through explicit capability and availability state.\n\n---\n\n## Phase 28.x - Recording Query Architecture\n\nGoal:\n\nModernize recording read access with a dedicated recording query architecture.\n\nPlanned direction:\n\n- define recording query requests and query scopes\n- keep real recordings owned by VDR\n- keep SQLite metadata separate from VDR recording truth\n- prepare backend-aware recording reads\n- prepare filtered and paginated recording lists\n- keep metadata enrichment optional and layered above real recording data\n- keep heavy recording-related data optional where needed\n- avoid coupling recording queries directly to frontend-specific views\n\nArchitecture rule:\n\n```text\nVDR recording state is authoritative.\nVDR-Suite metadata is complementary.\n```\n\nExpected boundary:\n\n```text\nRecording REST API -> Recording query service -> VdrService / IVdrAdapter\n                                      -> VDR-Suite metadata repositories\n```\n\nSQLite remains appropriate for:\n\n- recording metadata owned by VDR-Suite\n- workflow state\n- job state\n- user state\n- optional indexes\n\nSQLite must not become the source of truth for real VDR recordings.\n\n---\n\n## Phase 29.x - Recording REST API Modernization\n\nGoal:\n\nStabilize recording REST contracts for future clients.\n\nPlanned direction:\n\n- introduce client-safe recording response contracts\n- add filtering and pagination where needed\n- preserve backend identity in multi-backend responses\n- expose metadata-enriched views without hiding original VDR recording data\n- keep API behavior frontend-independent\n- make error models predictable\n- make capability information visible where actions or enriched views depend on backend support\n\nArchitecture rule:\n\nRecording REST responses must remain stable for clients while preserving the distinction between VDR-owned recording data and VDR-Suite-owned metadata.\n\n---\n\n## Phase 30.x - Recording Actions\n\nGoal:\n\nDefine recording operations as explicit action and job boundaries.\n\nPlanned direction:\n\n- model operations such as move, rename, cut, repair and delete as actions\n- route long-running operations through job boundaries\n- keep destructive operations explicit and auditable\n- preserve backend capability checks\n- avoid direct frontend-to-filesystem action coupling\n- prepare dry-run or validation behavior where useful\n- separate synchronous validation from asynchronous execution\n\nArchitecture rule:\n\n```text\nClient request -> Action boundary -> Job boundary -> backend/platform adapter\n```\n\nRecording actions must not bypass permission checks, backend capability checks or job lifecycle tracking.\n\n---\n\n## Phase 31.x - Recording Metadata and Job Integration\n\nGoal:\n\nIntegrate recording metadata, workflow state and jobs without making VDR-Suite the owner of real recordings.\n\nPlanned direction:\n\n- connect recording metadata to recording query results\n- connect recording actions to job lifecycle state\n- track workflow state in SQLite\n- support scraper-derived metadata as optional enrichment\n- keep metadata, posters and preview data as heavy or optional domains\n- expose job state in a client-safe way\n- keep metadata refresh behavior explicit\n\nArchitecture rule:\n\nMetadata and jobs enrich VDR-owned recording data. They must not replace VDR as the authoritative source for real recording existence, location or lifecycle.\n\n---\n\n## Phase 32.x - Multi-VDR Federation\n\nGoal:\n\nExpose multiple VDR backends through one coherent API model.\n\nPlanned direction:\n\n- make multiple backend nodes visible to clients\n- preserve backend identity in all federated read models\n- support default backend behavior where a single backend is configured\n- support backend-specific reads for multi-backend environments\n- prepare cross-backend summaries without hiding backend boundaries\n- keep backend capability information visible\n- avoid unsafe implicit merging of recordings, timers or EPG data\n- prepare federation for later permission-aware access\n\nArchitecture rule:\n\n```text\nFederated API -> Backend registry / runtime contexts -> per-backend VDR services\n```\n\nMulti-VDR federation must not make separate VDR systems look like one filesystem or one timer namespace unless an explicit higher-level feature defines safe merge semantics.\n\n---\n\n## Phase 33.x - Permission and Role System\n\nGoal:\n\nAdd permission-aware access above the multi-VDR foundation.\n\nPlanned direction:\n\n- introduce roles and permissions for API access\n- support backend-specific rights\n- support read-only and view-only backend modes\n- protect recording actions behind explicit permissions\n- protect destructive actions behind stricter permissions\n- store VDR-Suite-owned permission state in SQLite\n- keep authorization independent from a specific frontend\n- prepare API responses that can expose capability and permission state safely\n\nExample target:\n\n```text\nBackend A: editable\nBackend B: view-only\n```\n\nArchitecture rule:\n\nPermissions apply above backend discovery and before action execution. A backend may be technically capable of an operation while the current user or client role is not allowed to perform it.\n\n---\n\n## Phase 34.x - Advanced Search\n\nGoal:\n\nIntroduce advanced search across recordings, selective EPG views and metadata after the read boundaries are stable.\n\nPlanned direction:\n\n- search recording metadata and VDR recording views\n- search selective EPG windows without requiring a persistent full EPG mirror\n- use optional SQLite indexes only for VDR-Suite-owned or explicitly indexed data\n- keep heavy-domain query behavior explicit\n- avoid SearchTimer coupling in this phase\n- support backend-aware search results\n- preserve original backend identity in search responses\n- separate interactive search from scheduled search behavior\n\nArchitecture rule:\n\nAdvanced search may combine VDR-owned data and VDR-Suite-owned metadata, but search indexes must not silently become the authoritative source for VDR state.\n\n---\n\n## Phase 35.x - SearchTimer Architecture\n\nGoal:\n\nDefine SearchTimer support as backend-neutral and capability-based architecture.\n\nPlanned direction:\n\n- detect backend SearchTimer capabilities\n- support plugin-specific behavior through adapters\n- keep SearchTimer contracts separate from generic EPG reads\n- avoid making EPG persistence a prerequisite\n- preserve compatibility with backends that do not support SearchTimers\n- separate SearchTimer definition, validation, execution and result views\n- keep backend-specific plugin details out of frontend contracts where possible\n\nArchitecture rule:\n\n```text\nSearchTimer API -> capability-aware service -> backend/plugin adapter\n```\n\nSearchTimer support must be optional. It must not make EPG persistence mandatory and must not weaken the backend-neutral EPG query boundary introduced earlier.\n\n---\n\n## Phase 36.x - Windows Client API Stabilization\n\nGoal:\n\nStabilize API behavior for a future Windows client.\n\nPlanned direction:\n\n- harden REST response contracts\n- keep API models frontend-independent\n- preserve backend identity and permission information\n- ensure recording, EPG, metadata, job and action APIs are predictable\n- avoid client-specific backend shortcuts\n- document stable request and response shapes\n- keep transport-specific behavior separated from domain REST contracts\n- prepare API compatibility expectations for later Web, Android, iOS and OSD clients\n\nArchitecture rule:\n\nThe Windows client must consume stable VDR-Suite API contracts. It must not depend on RESTfulAPI-specific URLs, local filesystem details or undocumented backend behavior.\n\n---\n\n## Future Product Layers\n\nPossible later layers:\n\n- Web frontend\n- Windows frontend\n- Android frontend\n- iOS frontend\n- OSD integration\n- stream provider integration\n- media library expansion\n- authentication and authorization\n- permission-aware backend federation\n\n---\n\n## Roadmap Rule\n\nThis file describes direction, not detailed implementation history.\n\nCompleted implementation detail belongs in [Completed Phases](../development/completed-phases.md).\n\n---\n\n## Back\n\n- [Back to README](../../README.md)\n- [Back to Documentation Index](../index.md)\n- [Back to Project Overview](../project-overview.md)\n\n
