current domain set.\n- Snapshot read APIs are available for status, channels, timers, events and recordings.\n- Snapshot cache, snapshot access and partial refresh planning are in place.\n- Snapshot cache generation tracking is implemented in `SnapshotCacheService`.\n- Snapshot change feed service, serializer and read-only REST controller are implemented.\n- Snapshot change feed entries can be appended to an existing runtime-owned feed.\n- Daemon runtime owns the snapshot change feed and updates it after VDR polling.\n- Runtime diagnostics are integrated through structured runtime measurement boundaries.\n- Backend identity is present in snapshot change feed entries, snapshot read metadata and cached snapshots.\n- `BackendNode` and `BackendRegistry` provide the backend identity foundation.\n- `BackendRegistryService`, `BackendRegistryJsonSerializer` and `BackendRegistryController` expose the registry through service and REST boundaries.\n- `ApiRouter` exposes backend registry routes, backend-aware snapshot routes and the multi-backend snapshots route.\n- `SnapshotCache` can store snapshots per backend while preserving the legacy single-snapshot interface.\n- `SnapshotAccessService` resolves snapshots through the multi-backend cache and can return all cached backend snapshots.\n- `VdrSnapshotReadService` supports backend-aware reads and multi-backend snapshot list reads.\n- `VdrSnapshotReadJsonSerializer` can serialize multi-backend snapshot summary lists.\n- `VdrController` exposes default VDR reads, backend-specific reads and multi-backend snapshot summary reads.\n- `ApiRouter` regression coverage verifies the multi-backend snapshots REST route.\n- `VdrSnapshotBuilder` can assign a stable backend ID to generated snapshots.\n- `PollingService` and `BackendPollingCoordinator` support backend-aware polling coordination.\n- `DaemonRuntime` creates backend runtime contexts from the backend registry.\n- Future multi-VDR runtime configuration can build on the registry-driven context creation model.\n- Backend identity, federation, capability and lifecycle strategy are documented through ADRs.\n- `VdrEventQuery` provides the first backend-neutral selective EPG query contract.\n- `IVdrAdapter` and `VdrService` support query-based event access while preserving legacy full-event access.\n- `RestfulApiVdrAdapter` maps selective event queries to existing RESTfulAPI query parameters.\n`DomainRefreshPolicy` classifies Events / EPG as a heavy domain.\n`SnapshotRefreshPlanner` no longer creates automatic full EPG refresh work for EventsChanged.\n\n---\n\n## Selective Backend Query Rule\n\nVDR-Suite should prefer selective backend queries over full-domain transfers whenever possible.\n\nHeavy domains must not use full-domain runtime refreshes as the default strategy.\n\nHeavy domains currently include:\n\n- EPG\n- metadata\n- posters\n- fanart\n- preview data\n- scraper-derived data\n\nPreferred runtime strategies are:\n\n- channel-scoped queries\n- time-window queries\n- object-specific queries\n- change-hint driven refreshes\n\nPerformance goal:\n\nBackend workload should remain comparable to established VDR frontends such as live whenever equivalent information is requested.\n\n---\n\n## Current Implemented API Areas\n\nSnapshot-backed default VDR read APIs:\n\n```text\nGET
